Charter v1.0.0-draft

A portable, AI-readable, machine-validatable design system contract.

In one line: a DESIGN.md you can fail the build on.

AI agents are good at writing code that looks right. They are reliably bad at writing code that is right — that uses the design system's tokens, respects the design system's components, and behaves the way the design system was meant to behave.

A Charter is a single file at the root of your project that declares what your design system permits — in a form a model can read at generation time and a linter can refuse at audit time.

Quick look

# CHARTER.yaml
$schema: https://usecharter.dev/v1/charter.schema.json
name: Hello
version: 0.1.0

tokens:
  color:
    primary: { $value: "#0066ff", $type: color }
    surface: { $value: "#ffffff", $type: color }
    text:    { $value: "#1f2937", $type: color }

components:
  registry: "@hello/ui"
  allowed: [Button]

rules:
  hardcoded_colors: error

That is a Charter. A linter pointed at this file will reject any generated code that uses a color outside the three declared tokens. The design system is no longer a suggestion.

What you get

  • Tokens in W3C DTCG format — interoperable with Style Dictionary, Figma Variables, and DESIGN.md
  • Component allowlists — declare what UI may compose with
  • Rules with graduated severity (error / warn / off)
  • Profiles — different rules for prototype vs production
  • Notes — free-text rationale, preserved across tools
  • A JSON Schema any standards-compliant validator can use

Why it exists

A "falsifiable spec" — one specific enough that an automated check can refuse output that violates it — produces measurably better AI output. But every line in a good falsifiable spec ("no scale-105", "do NOT hand-roll <div className="rounded-lg border…">") comes from a scar the writer has already earned. A PM or designer writing their first AI spec has none of those scars. They cannot write that spec.

That's a tooling failure dressed as a discipline problem. A Charter shifts the cost of precision from "scars accumulated by hand" to "tool installed once."

What's measured, what isn't

Rule defaults in the spec are tagged [supported] (backed by measurement) or [preliminary] (educated guess, measurement pending). The most consequential structural claim — that constraining AI to a component allowlist reduces drift — is backed by a five-test validation arc with 36+ blind-scored builds. Specific numeric thresholds (like delta_e_threshold: 2.0) are preliminary and tracked in a public measurement plan for closing before v1.0.0 final.

Reviewers are invited to challenge any of it.
